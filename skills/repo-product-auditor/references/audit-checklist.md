# Repo Product Audit Checklist

Use this as the detailed reference checklist for repo audits.

## 1. First impression

- Is the repo name understandable?
- Is the one-line description specific?
- Can a visitor understand the value in 10 seconds?
- Does the repo look active?
- Is the target user obvious?

## 2. README hero

A strong README hero should include:

- Clear title
- One-sentence positioning
- Status badges
- Screenshot, GIF, or terminal preview
- Quick value statement
- Primary call-to-action

Preferred structure:

```markdown
# Project Name

One clear sentence explaining what this does, for whom, and why it matters.

[Badges]

Screenshot / terminal preview

## Why this exists
## Quick start
## Features
## Use cases
## Architecture
## Roadmap
```

## 3. Product positioning

Check whether the repo answers:

- What is it?
- Who is it for?
- What problem does it solve?
- Why is it better than a script, note, or gist?
- What is the most impressive use case?

## 4. Visual proof

Look for:

- Screenshots
- Terminal output
- GIF demo
- Architecture diagram
- GitHub Pages demo
- Before/after examples

If visual proof is missing, recommend it early.

## 5. Setup clarity

Check whether a new visitor can run it without guessing:

- Requirements
- Install command
- Run command
- Example output
- Troubleshooting
- Supported platforms

## 6. Discovery

Check:

- GitHub topics
- Description field
- README keywords
- Pinned repo position
- Release tags
- Social links
- Demo link
- Case page

## 7. Portfolio value

Ask:

- Does this repo show judgment?
- Does it show architecture thinking?
- Does it show operational realism?
- Does it connect to the author's professional identity?
- Would a recruiter, architect, or technical peer understand why this matters?

## 8. Recommended priority

Use this order:

1. Positioning
2. README hero
3. Visual proof
4. Quick start
5. Case page
6. GitHub topics
7. Release polish
8. Social/posting strategy
